# Agent 共通ガイドライン

## このファイルについて

このファイルはエージェントの振る舞いを定める。
リポジトリ固有の詳細情報は `docs/` に委ねる（`docs/`の索引はルートの`README.md`を参照）

エージェントから見て、Git にない知識は存在しないのと同じである。
再利用すべき判断・学んだ事実は、必ず `docs/` またはコードに残すこと。

## 言語

会話・レビュー・コメント・ドキュメントは**日本語**。
専門用語は短く補足する

## あなたの作業の流れ

1. Plan:

- `docs/`の関連ファイルを確認する

2. Implement:

- コード変更は 小さく、レイヤーに収まる単位で 行う。1 PR = 1 論理的変更。

3. Verify:

- 以下がグリーンであることを確認するまで決して完了としない
  - `pnpm check`
  - `pnpm test`

4. Document Update：

- 振る舞いやアーキテクチャに影響する変更は docs/ の該当ファイルを 同じPR内で 必ず更新する
- 設計判断（採用/不採用、トレードオフ）を伴う変更は [`docs/adr/`](./docs/adr/) に ADR を追加または更新する

5. PR作成:

- worktree で作業している場合、実装が完了したらコミット → プッシュ → PR 作成まで一連で実行する。
- ユーザーが「PR を作って」と指示してくるのを待つ必要はない。

6. PR レビュー & マージ:

- PR 作成後、`gh pr review --approve` でセルフ承認し、`gh pr merge --merge --delete-branch` でそのままマージする。
- ユーザーが「マージして」と指示してくるのを待つ必要はない。

## 何を信頼して正しさを検証するか？(フィードバックセンサー)

### 強く信頼してよいもの: 機械が判定する結果（計算的センサー）

linter、type checker、test の結果は、**決定論的(同じ入力なら同じ結果)**で嘘をつかない。違反があればまず信じて修正する。

- テストが落ちたら、必ず根本原因を修正する。
- テストの無効化、catch で握りつぶす等の暫定回避は禁止。

### 参考程度にとどめるもの: AI が生成した情報（推論的センサー）

AI が書いた PR サマリ、コミットメッセージ、レビューコメントなどは、もっともらしく見えても**確率的で、間違いを含むことがある**。

- 上記の「機械が判定する結果」と矛盾した場合は、必ず機械の方を信じる。
- 重要な判断の前には、要約ではなく元情報(コード、ログ、ファイル)を直接確認する。

### そもそも検証手段が存在しないもの（欠落センサー）

「ユーザーが本当に求めている価値を満たしているか」の最終的な検証は、機械にもAIにもできない。

- タスクの受け入れ条件が曖昧な場合、実装に進む前に Plan で明示的に質問する。
- 情報を「推測」で補うことは禁止。

## コンテキスト衛生

- 新しい知識はリポジトリに残す:
  - 作業中に学んだ非自明な事実(API の癖、ライブラリのバグ、合意された設計判断)は必ず docs/ に書き残す。
- サブエージェントを使ってコンテキストを分離する
  - **使うべき場面**: 大量のファイルを読む探索・調査(結論だけ親に返す)、独立して並列実行できるタスク、レビュー・検証(自己評価バイアスの構造的回避)
  - **使うべきでない場面**: 横断的なリファクタ、設計判断を含む実装(密な文脈共有が必要なため、要約で渡すと情報が落ちる)
